# Novel2Script YAML Studio 需求分析文档

## 1. 项目背景

本项目参加 72h AI 应用开发比赛，选题为「AI 小说转剧本工具」。

题目要求开发一款 AI 辅助剧本创作工具，将 3 个章节以上的小说文本自动转换为结构化剧本 YAML 格式，让作者快速获得可编辑、可继续打磨的剧本初稿。题目还要求额外编写 YAML Schema 文档，并说明 Schema 的设计原因。

本项目第一版不做一个普通的“AI 改写器”，而是定位为「小说 IP 结构化改编工作台」：先从小说中抽取故事资产，再根据用户选择的改编目标，生成符合结构约束的剧本 YAML 初稿。

## 2. 目标用户

### 2.1 主要用户

小说作者、网文作者。

他们已经拥有小说原文，但在将小说改编成剧本时，需要花大量时间整理人物、关系、世界观、关键事件和章节内容。第一版产品优先服务这类用户，因为他们最贴近题目要求，也最容易通过 demo 展示完整用户价值。

### 2.2 次要用户

短剧创作者、漫剧创作者。

他们更关注小说内容能否被快速拆解为场景、动作、对白和分镜。第一版通过短剧剧本 YAML 和简化 `shots` 分镜字段，帮助这类用户快速获得可继续加工的剧本初稿。

### 2.3 第一版暂不主打的用户

小说推文创作者、视频生成创作者、二创模板创作者。

这些方向有创意，但容易让第一版偏离“剧本工具”的主线。第一版只把它们放入后续迭代方向，不作为必须交付范围。

## 3. 用户痛点

### 3.1 小说改编耗时

3 章以上小说通常包含多个人物、关系、场景、关键事件和情绪变化。作者如果手动整理这些信息，再改写成剧本结构，需要大量重复劳动。

### 3.2 结构化改编门槛高

小说是叙事文本，剧本更强调场景、动作、对白、冲突和镜头。很多作者有故事内容，但不一定熟悉如何把长篇叙事拆成可拍、可演、可继续编辑的剧本结构。

### 3.3 普通 AI 改写不可控

直接让 AI 改写小说，容易出现凭空添加剧情、改变人物动机、丢失人物关系、忽略原文关键事件等问题。作者需要知道 AI 生成内容来自原文哪里，才能判断是否乱改。

### 3.4 不同题材和媒介风格难快速适配

悬疑、都市、玄幻、言情等题材的表达重点不同；短剧和漫剧的节奏也不同。作者很难快速判断自己的小说适合哪种改编策略和表达方式。

### 3.5 生成结果难复用

普通 AI 输出通常是一段文本，后续很难校验、编辑、下载、复用或接入其他创作流程。结构化 YAML 可以作为可编辑的剧本资产包，为后续分镜、模板、视频提示词等能力留下扩展空间。

## 4. 产品定位

Novel2Script YAML Studio 是一个面向小说作者和短剧/漫剧创作者的 AI 结构化改编工具。

它将 3 章以上小说文本拆解为人物、关系、世界观和关键事件，并生成符合 YAML Schema 的短剧剧本与简化分镜初稿，帮助作者节省改编时间、降低剧本创作门槛，并通过来源追溯和 Schema 校验减少 AI 乱改问题。

## 5. 差异化价值

### 5.1 不是简单改写，而是故事资产抽取

普通 AI 改写工具通常是“输入小说，输出一段文本”。本项目先抽取人物、关系、世界观和关键事件，再生成剧本结构，让作者能看到小说内容如何被拆解和重组。

### 5.2 Schema-first 的结构化输出

本项目要求 AI 生成符合结构约束的 YAML，而不是自由文本。结构化输出更容易校验、复制、下载和继续编辑，也更适合后续接入分镜、模板或视频工作流。

### 5.3 来源追溯降低乱编风险

第一版会设计 `source_refs` 来源追溯概念，让生成的场景或关键内容能够标记来自原文的章节或情节依据。这样作者可以检查 AI 是否偏离原文。

### 5.4 改编配置让输出更可控

第一版轻量支持题材类型和改编策略选择，让用户能影响生成方向。例如选择悬疑、都市、玄幻、言情、通用题材，或选择忠实改编、压缩改编、冲突强化等策略。

## 6. 第一版 MVP 范围

第一版目标是完成一个本地可运行、可演示、可解释的闭环：

```text
输入 3 章以上小说文本
-> 抽取故事资产
-> 选择题材和改编策略
-> 生成短剧剧本 YAML
-> 展示 YAML 与简化剧本/分镜预览
-> Schema 校验
-> 复制或下载 YAML
```

### 6.1 Must have

- 支持输入 3 章以上小说文本。
- 自动抽取故事资产：人物、人物关系、世界观、关键事件。
- 生成短剧剧本 YAML。
- YAML 中包含简化 `shots` 分镜字段，用于基础分镜预览。
- 支持 `source_refs` 来源追溯概念。
- 支持 YAML 预览。
- 支持 Schema 校验结果展示。
- 支持复制 YAML。
- 支持下载 YAML。
- 提供 Mock 示例兜底，保证没有真实 AI API key 时也能演示完整流程。

### 6.2 Light have

- 题材选择：悬疑、都市、玄幻、言情、通用。
- 改编策略：忠实改编、压缩改编、冲突强化。
- 简单剧本预览。
- 简单分镜预览。

这些能力第一版只做轻量配置和展示，不做复杂编辑器。

### 6.3 Later

- 完整小说推文模式。
- 通用戏剧模板库。
- 角色原型库。
- 抽卡式二创。
- 视频生成提示词导出。
- 用户自定义模板。
- 模板商店、搜索、收藏。
- 登录、数据库、云同步。
- 复杂剧本编辑器。

## 7. 第一版不做什么

第一版明确不做以下内容：

- 不做完整小说推文生成器。
- 不做模板库和角色原型库。
- 不做抽卡二创。
- 不做视频生成提示词导出。
- 不做商店、搜索、收藏。
- 不做用户登录。
- 不做数据库保存。
- 不做云端同步。
- 不做复杂可视化编辑器。
- 不直接接入视频生成模型。

这些能力可以作为第二版或未来计划写入 README 或后续文档，但不能进入第一版必须交付范围。

## 8. 核心用户流程

1. 用户打开本地应用。
2. 用户粘贴 3 章以上小说文本，或使用示例文本。
3. 用户选择题材类型。
4. 用户选择改编策略。
5. 用户点击生成。
6. 系统抽取故事资产，包括人物、关系、世界观和关键事件。
7. 系统生成短剧剧本 YAML。
8. 系统展示 YAML 预览。
9. 系统展示简化剧本/分镜预览。
10. 系统展示 Schema 校验结果。
11. 用户复制或下载 YAML。

## 9. 功能需求

### 9.1 小说输入

用户可以输入 3 章以上小说文本。系统需要能够判断输入是否为空，以及章节数量是否满足要求。

验收标准：

1. 用户输入 3 章以上小说文本后，可以继续生成。
2. 用户没有输入内容时，系统显示错误提示。
3. 用户输入少于 3 章内容时，系统显示章节不足提示。
4. 输入错误不会导致页面崩溃。

### 9.2 故事资产抽取

系统从小说文本中抽取人物、人物关系、世界观和关键事件，用于后续剧本生成。

验收标准：

1. 用户输入有效小说文本并点击生成后，系统返回人物信息。
2. 系统返回人物关系信息。
3. 系统返回世界观或背景设定摘要。
4. 系统返回关键事件列表。
5. 抽取失败时，系统显示错误提示。

### 9.3 改编配置

用户可以选择题材类型和改编策略，影响最终剧本 YAML 的生成方向。

验收标准：

1. 用户可以选择题材类型。
2. 用户可以选择改编策略。
3. 生成结果中能体现用户选择的题材或策略。
4. 用户不选择时，系统使用默认配置。

默认配置：

- 题材类型：通用。
- 改编策略：忠实改编。

### 9.4 剧本 YAML 生成

系统根据小说文本、故事资产和改编配置，生成短剧剧本 YAML。

验收标准：

1. 用户输入有效小说文本后，可以生成 YAML。
2. YAML 至少覆盖项目、来源、故事资产、剧本内容、元信息等核心模块。
3. YAML 包含人物、关系、场景、分镜、对白、动作和来源追溯信息。
4. 生成失败时，系统显示错误提示。
5. 没有真实 AI API key 时，系统仍可使用 Mock 示例完成演示。

### 9.5 YAML 预览与 Schema 校验

系统展示生成的 YAML，并显示基础 Schema 校验结果。

验收标准：

1. 用户生成 YAML 后，可以在页面查看 YAML 内容。
2. 系统显示 Schema 校验通过或失败状态。
3. YAML 校验失败时，系统显示可理解的错误提示。
4. 校验失败不会导致页面崩溃。

说明：完整字段契约见 [YAML Schema 设计文档](schema.md)。第一版的 Schema 校验以该文档中的核心模块、关键字段和基础校验原则为依据。

### 9.6 剧本与分镜预览

系统根据 YAML 展示简化剧本预览和基础分镜预览，帮助用户快速理解生成效果。

验收标准：

1. 用户生成 YAML 后，可以看到场景列表。
2. 用户可以看到每个场景的简要内容。
3. 用户可以看到基础分镜信息，例如画面描述、人物动作或对白摘要。
4. 预览内容与 YAML 内容保持一致。

### 9.7 复制与下载

用户可以复制或下载生成的 YAML。

验收标准：

1. 用户点击复制后，可以复制 YAML 内容。
2. 用户点击下载后，可以获得 YAML 文件。
3. 没有生成结果时，复制和下载操作不会导致页面崩溃。

### 9.8 Mock Demo 兜底

系统提供示例输入和示例输出，保证没有真实 AI API key 时也能本地演示完整流程。

验收标准：

1. 用户可以使用示例小说文本快速填充输入。
2. 在没有真实 AI API key 的情况下，系统可以返回 Mock YAML。
3. Mock YAML 可以展示 YAML 预览、校验结果、剧本预览和下载流程。

## 10. YAML Schema 核心模块规划

本需求文档只规划 YAML Schema 的核心模块。字段级契约、字段作用、是否必填和设计原因见 [YAML Schema 设计文档](schema.md)。

第一版的 Schema 文档主要作为后续 AI 生成、Mock 示例、YAML 校验、剧本预览和基础分镜预览的字段契约。`shots` 只作为第一版基础分镜预览字段，不扩展为完整视频生成系统。

第一版 YAML 需要围绕以下模块设计：

- `project`：项目和改编任务的基础信息。
- `source`：原小说输入来源和章节信息。
- `story_bible`：故事资产集合。
- `characters`：人物设定。
- `relationships`：人物关系。
- `worldbuilding`：世界观和背景设定。
- `key_events`：关键事件。
- `adaptation`：题材类型和改编策略。
- `script`：生成的剧本内容。
- `scenes`：场景。
- `shots`：简化分镜。
- `dialogues`：对白。
- `actions`：动作或表演提示。
- `metadata`：生成信息、警告和校验状态。
- `source_refs`：来源追溯。
- `warnings`：生成或校验中的风险提示。

## 11. 验收标准

第一版整体完成时，需要满足以下验收标准：

1. 用户可以输入 3 章以上小说文本。
2. 用户输入少于 3 章内容时，系统显示章节不足提示。
3. 用户可以选择题材类型和改编策略。
4. 用户点击生成后，系统返回结构化剧本 YAML。
5. YAML 包含故事资产、剧本场景、简化分镜、对白、动作和来源追溯信息。
6. 页面可以展示 YAML 预览。
7. 页面可以展示简化剧本/分镜预览。
8. 页面可以展示 Schema 校验结果。
9. 用户可以复制 YAML。
10. 用户可以下载 YAML。
11. 没有真实 AI API key 时，用户仍可通过 Mock 示例跑通完整演示流程。
12. 生成失败、校验失败、输入错误时，页面不会崩溃，并显示可理解的错误提示。

## 12. 评审规则映射

### 12.1 作品完整度与创新性

对应评分项：产品设计合理性、功能完整度、交互流畅度、是否新颖有创意。

本项目通过以下方式体现：

- 完整闭环：输入小说、生成 YAML、校验、预览、复制、下载。
- 产品设计合理：围绕小说作者和短剧/漫剧创作者的真实改编流程设计。
- 创新性：不是普通 AI 改写，而是“小说 IP 结构化改编工作台”。
- 结构化价值：通过 YAML 将故事资产、剧本和分镜组织成可编辑资产包。
- 可控性：通过题材、策略和来源追溯降低 AI 乱改风险。

### 12.2 开发过程与质量

对应评分项：架构清晰度与合理性、代码健壮度、规范、可读性、PR 数量与质量、commit 分布合理性。

本项目通过以下方式体现：

- 每个 PR 只做一个明确目标。
- README 保持短小，深层产品设计放入 `docs/`。
- 后续 Schema 文档单独成 PR。
- 功能实现保留 Mock 兜底，保证本地可演示。
- 输入错误、生成失败、校验失败都需要有清晰提示。
- 不引入登录、数据库、复杂编辑器等 72h 内难以解释的设计。

### 12.3 演示与表达

对应评分项：demo 视频清晰完整表达作品功能和效果。

Demo 应重点展示：

1. 小说作者改编耗时、结构混乱、AI 易乱改的痛点。
2. 输入 3 章以上小说文本。
3. 选择题材和改编策略。
4. 生成短剧剧本 YAML。
5. 展示故事资产、场景、分镜、对白和来源追溯。
6. 展示 Schema 校验结果。
7. 复制或下载 YAML。
8. 说明后续可扩展到模板库、推文模式和视频提示词。

## 13. 第二版迭代方向

第二版可以围绕“二创”和“多媒介改编”扩展，但不进入第一版必须范围。

可考虑方向：

- 小说推文模式。
- 通用戏剧模板库，例如雨夜对峙、身份反转、主角黑化、审讯室心理战。
- 角色原型库，例如冷静复仇者、疯批反派、白切黑女主。
- 自定义模板，让作者基于已有 YAML 剧本继续二创。
- 更完整的分镜和视频生成提示词导出。
- 接入 AI 视频工作流。

第二版必须继续避免直接使用具体版权 IP、角色名、台词或桥段。模板应描述通用戏剧结构，而不是复刻具体作品。
